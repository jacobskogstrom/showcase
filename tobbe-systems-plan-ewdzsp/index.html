🔌 The MCP trick (the real unlock)

Tickets live behind MCP, so any AI chat — Cursor, Claude, anywhere — can file or fetch them. "Put this in a ticket in bionicbrain" from one chat; weeks later in another repo: "find the ticket and implement it". No copy-paste, no lost context.

core pattern

🧾 Ticket anatomy

Every ticket has: Goal · Behaviour · UI · API/MCP · Open design questions · Acceptance criteria · Notes/relations. Plus "Out of scope / later" and even "Ops lessons" — the ticket doubles as living documentation after shipping.

Tickets reference each other by number — approval flow "depends on audit log (#8)", "relates to BeanStalk (#5), SSO (#1)". A tiny dependency graph across the backlog, so agents see context when they pick a ticket up.

→

Why it works: the ticket is the contract. Specification (open questions) forces decisions before building, and acceptance criteria give the agent a testable "done". That's the professional development approach you're after — not the tool, the discipline.

Swedish accounting — reality check before building

⚖️ The baseline law

Every AB is bokföringsskyldig under Bokföringslagen: one verifikation per affärshändelse, recorded chronologically + systematically, archived 7 years. Corrections happen via new rättelseverifikationer — never edits. Tobbe's "immutable posts" is law, not taste.

bokföringslagen

📐 K2 + kontantmetoden fit SC AB

K2 is the simplified reporting framework for smaller ABs. Kontantmetoden (bokslutsmetoden) is allowed under ~3 MSEK turnover — book when money moves, settle the rest at bokslut. Tobbe chose both; SC AB likely should too. Verify against SC AB's registered räkenskapsår.

verify with skatteverket

📋 Self-built system = extra duties

BFN's bookkeeping rules require systemdokumentation and behandlingshistorik for the bookkeeping system itself — how it works and what it did. A self-built system must document itself. This is Q10 to Tobbe, and non-negotiable in any build.

often missed

🏛️ The last mile is manual anyway

Årsredovisning to Bolagsverket yearly regardless of size (small ABs usually need no revisor below the thresholds), momsdeklaration to Skatteverket on schedule. No self-built system files these for you — plan for an accountant to sanity-check the first bokslut whatever you build.

budget for it
Build vs buy, honestly: Fortnox (~a few hundred kr/month) or Bokio outsource law changes, moms rules, and filing formats — the boring risk. Building gives you MCP-native accounting agents can drive (receipt → draft verifikation → human approves), a real dogfooding story for the AI-operator brand, and Tobbe as a proven reference. There's also a middle path worth pricing: Fortnox + a thin MCP wrapper over its public API — compliance outsourced, agent workflow kept. Decide at the Phase 2 gate, after Tobbe's answers to Q9, Q14 and Q17.
